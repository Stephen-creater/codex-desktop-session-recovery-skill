## Codex Desktop 聊天记录“消失”了，但大概率并没有真的丢

很多人看到 Codex Desktop 项目侧边栏出现 `No chats` 的第一反应是：完了，聊天记录没了。

这次我把问题拆到底层看了一遍，结论比直觉更具体：

1. 很多时候，底层数据还在。
2. 真正出问题的是“可见性”和“归类”。
3. 至少有三类故障会让你误以为聊天被删了。

### 第一类：Recent 窗口把旧项目挤没了

如果一个新项目最近很活跃，它可能占掉大量最近线程名额。旧项目虽然仍在 `~/.codex/state_5.sqlite` 里，但没被侧边栏加载出来，看起来就像消失。

### 第二类：线程被误标成 projectless

这类问题更隐蔽。线程明明有外部 `cwd`，但全局状态却把它放进了 `projectless-thread-ids`，再配一个宽泛的根路径提示。这样一来，项目视图就找不到它。

### 第三类：路径身份变了

Codex Desktop 的线程和项目关联依赖 `cwd`。如果更新、迁移或根路径变化让这个 `cwd` 身份不再一致，历史线程就会“还在本地，但不再出现在对应项目下”。

## 这套修复方案做了什么

我没有去改官方安装包，而是做了一套本地可复用修复流程：

- 先只读审计
- 再 dry-run 预演
- 最后只改 `.codex-global-state.json`
- 不碰原始聊天内容
- 不删 `sessions/` JSONL

修复动作本质上只有两类：

- 把明显误标的 projectless 线程重新归回项目线程
- 按现有线程的精确 `cwd` 重建项目根和显示顺序

## 什么时候该自己修，什么时候该找官方

如果 SQLite 行和 JSONL 文件都还在，本地可修概率很高。

如果数据库迁移已经损坏、app-server 都起不来，或者修完以后 GUI 依然不认精确 `cwd`，那就该把证据链整理好，上报官方了。
